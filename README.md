# Фабрика гипотез

Интеллектуальная система генерации и прозрачного ранжирования проверяемых
научно-исследовательских гипотез для снижения потерь цветных металлов с
хвостами обогатительных фабрик (кейс «Задача 1. Фабрика гипотез»,
AI-хакатон Норникель).

## Словарь: что такое «отчёт института по хвостам»

**Хвосты** — отвальные продукты флотационного обогащения руды: то, что после
извлечения концентрата уходит в хвостохранилище. Металл, попавший в хвосты,
считается потерянным, поэтому фабрики стремятся минимизировать его содержание.

**Отчёт института по хвостам** — Excel-файл с комплексным анализом хвостов
одной фабрики за период (примеры — в `data/task1/Пример N/Хвосты *.xlsx`):
- сколько руды переработано и сколько хвостов образовано (в сухих метрических
  тоннах, СМТ);
- содержание и масса потерянного никеля («Элемент 28») и меди («Элемент 29»);
- разложение потерь по **классам крупности** частиц: +125, −125 +71, −71 +45,
  −45 +20, −20 +10, −10 мкм (запись «−71 +45» = частицы мельче 71, но крупнее
  45 мкм);
- внутри каждого класса — **минералогия потерь**: в какой форме находится
  металл. Раскрытый Pnt/Cp — свободные зёрна пентландита/халькопирита (флотация
  их упустила); закрытый Pnt/Cp — сростки с породой (не раскрыты измельчением);
  примесь в пирротине и силикатная форма — текущей технологией не извлекаются;
  миллерит — извлекаемый сульфид никеля.

Именно по этой структуре диагностические правила системы определяют,
*где* и *почему* теряется металл, и что с этим можно сделать.

## Что делает система

**Вход:**
- отчёт института по хвостам (.xlsx: потоки хвостов × классы крупности ×
  минералогия потерь, «Элемент 28» = Ni, «Элемент 29» = Cu);
- целевой KPI и ограничения (свободным текстом);
- база знаний: учебники по флотации и обогащению (PDF), регламенты,
  исторические примеры гипотез (DOCX).

**Выход:** ранжированный список проверяемых гипотез, у каждой:
- формулировка в стиле мозговых штурмов фабрик;
- механизм влияния и обоснование с цитатами из литературы (книга + страница);
- ожидаемый эффект в тоннах металла и % от охваченных потерь;
- прозрачный разбор оценки по 5 критериям (ценность, реализуемость,
  новизна, обоснованность, проверяемость) с настраиваемыми весами;
- риски, дорожная карта лабораторной проверки, требуемые ресурсы;
- подграф графа знаний как «карточка доказательств».

## Архитектура

```
xlsx-отчёт ──► Парсер ──► Диагностика (интерпретируемые правила)
                              │  сигналы потерь: «2 088 т Ni в закрытых
                              │  сростках крупных классов → недораскрытие»
PDF/DOCX ──► Чанки ──► Векторный индекс (эмбеддинги Yandex, локально)
                │
                └────► Граф знаний (онтология флотации + LLM-извлечение
                        связей из литературы; JSON, без внешних БД)
                              │
        сигнал + подграф + цитаты ──► YandexGPT (GraphRAG) ──► гипотезы
                              │
              Ранжирование: 5 объяснимых критериев × веса эксперта
                              │
        Веб-UI / DOCX-отчёт / CSV / JSON-задачи (Jira/YouTrack) / фидбэк
```

Ключевые принципы:
- **Интерпретируемость.** Диагностика — явные правила с числами из отчёта;
  ранжирование — взвешенная сумма критериев, вклад каждого виден в карточке;
  LLM формулирует гипотезы строго по переданному контексту (GraphRAG) и
  обязана цитировать источники.
- **Локальность.** Векторный индекс — numpy на диске, граф — JSON. Наружу
  уходят только вызовы Yandex Foundation Models (в контуре заказчика
  заменяются на локальную LLM — интерфейс изолирован в `hypofactory/llm.py`).
- **Масштабируемость по доменам.** Предметные знания вынесены в
  `hypofactory/ontology.py` (узлы/связи/рычаги) и `diagnostics.py` (правила);
  подключение нового домена = новая онтология + правила, ядро не меняется.
- **Обучение на фидбэке.** Оценки экспертов (принять/отклонить) сохраняются
  (`artifacts/feedback.json`) и смещают рейтинг семантически похожих гипотез
  при пересчёте и в следующих запусках. На гипотезу хранится одна актуальная
  оценка: повторное нажатие другой кнопки заменяет вердикт.

## Критерии ранжирования и веса

Итоговый балл гипотезы — взвешенная сумма пяти независимых оценок (каждая 0–1),
вклад каждой виден в карточке гипотезы:

| Критерий | Что измеряет | Как считается |
|---|---|---|
| Ценность (0.30) | сколько металла в деньгах вернёт гипотеза | середина вилки ожидаемого эффекта (т) × цена металла, нормировано на лучшую гипотезу |
| Реализуемость (0.20) | цена и скорость внедрения | класс затрат рычага: настройка = 1.0, замена узла = 0.6, новый передел = 0.25 |
| Новизна (0.15) | отличие от уже испробованных решений | 1 − максимальная смысловая близость (эмбеддинги) к списку известных решений фабрик из кейса |
| Обоснованность (0.20) | сколько подтверждений | количество цитат из литературы + опора на узлы графа + наличие механизма |
| Проверяемость (0.15) | можно ли быстро проверить | полнота плана эксперимента + наличие критерия успеха |

Веса в скобках — значения по умолчанию (`hypofactory/config.py`,
`DEFAULT_WEIGHTS`): нейтральная стартовая настройка с упором на ценность,
принятая нами как разумный дефолт для приоритизации НИОКР. Это именно
настройка эксперта, а не выученные параметры: в веб-интерфейсе они меняются
слайдерами, сумма нормируется к 1, рейтинг пересчитывается мгновенно без
повторной генерации.

## Структура репозитория

```
hypofactory/          ядро
  config.py           ключи, пути, цены металлов, веса критериев
  llm.py              клиент Yandex AI Studio (ретраи, дисковый кэш, JSON-режим)
  tailings.py         парсер xlsx-отчётов по хвостам (устойчив к #REF!, пропускам)
  documents.py        PDF/DOCX → чанки с метаданными (файл, страница)
  vectorstore.py      локальный векторный индекс (numpy, косинусная близость)
  ontology.py         доменная онтология: минералы/операции/параметры/механизмы/рычаги
  graph.py            граф знаний: seed + LLM-извлечение триплетов, подграфы
  diagnostics.py      правила диагностики потерь → квантифицированные сигналы
  generator.py        GraphRAG-генерация гипотез (YandexGPT)
  ranker.py           прозрачное ранжирование + фидбэк экспертов
  pipeline.py         сквозной конвейер
  export.py           DOCX-отчёт, CSV, JSON-задачи
webapp/               FastAPI + одностраничный UI (карточки, граф на canvas)
scripts/
  build_kb.py         построение базы знаний (однократно)
  run_pipeline.py     CLI-прогон
artifacts/            построенная база знаний, кэш API, результаты
data/task1/           данные кейса (не в git)
```

## Запуск

```bash
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env   # вписать YC_API_KEY и YC_FOLDER_ID

# 1. Построить базу знаний по литературе кейса (один раз, ~10-20 мин)
python scripts/build_kb.py

# 2а. Веб-интерфейс
uvicorn webapp.main:app --host 0.0.0.0 --port 8000
# → http://localhost:8000

# 2б. Или CLI
python scripts/run_pipeline.py "data/task1/Пример 1/Хвосты КГМК.xlsx" \
    --out artifacts/result_kgmk
```

Без построенной базы знаний система тоже работает: гипотезы опираются на
затравочную онтологию (цитат из книг не будет).

## Проверка руками (по шагам)

В этом репозитории база знаний уже построена (`artifacts/`), поэтому:

```bash
# 0) активировать venv с зависимостями (см. «Запуск» выше)

# 1) офлайн-тесты: парсер, диагностика, граф — без API, секунды
python tests/test_pipeline_offline.py

# 2) CLI-прогон без веб-интерфейса (~1.5 мин; повторный запуск — из кэша)
python scripts/run_pipeline.py "data/task1/Пример 4/Хвосты ТОФ_2.xlsx" --out artifacts/check
# → консольная сводка + artifacts/check.html (открыть в браузере), .docx, .csv

# 3) веб-интерфейс
uvicorn webapp.main:app --port 8000
# → http://localhost:8000: выбрать пример, «Сгенерировать гипотезы»,
#   покрутить веса, «Пересчитать», вкладка «Граф знаний», экспорт
```

## Соответствие требованиям кейса

| Требование | Реализация |
|---|---|
| Приём разнородных данных | xlsx-отчёты, PDF-книги, DOCX; устойчивость к #REF! и пропускам |
| Извлечение сущностей и связей | онтология + LLM-извлечение триплетов из литературы |
| Выявление паттернов/пробелов | правила диагностики: формы потерь × классы крупности |
| Генерация гипотез | GraphRAG: сигнал + подграф + цитаты → проверяемое утверждение |
| Ранжирование | 5 объяснимых критериев, веса настраиваются в UI, вклад виден |
| Обоснование и цитирование | карточка доказательств: цитаты (книга, страница), подграф |
| Визуализация связей | интерактивный граф знаний (canvas) |
| Экспорт | DOCX-отчёт, CSV, JSON-задачи для Jira/YouTrack |
| Фидбэк экспертов | «Принять/Отклонить» в UI → поправка рейтинга похожих гипотез |

| Безопасность | локальное хранение; LLM-слой заменяем на on-prem модель |
| Мультиязычность | эмбеддинги Yandex мультиязычны; промпты — RU, литература RU/EN |
